# Event Contracts

Описание событий, которыми обмениваются сервисы через Kafka.

---

## Topic: `orders`

**Publisher:** `order-service`  
**Consumer:** `notifier`  
**Partition key:** `order_id` (string) — гарантирует порядок событий для одного заказа

---

### Общая обёртка

Все события имеют одинаковую структуру. Тип события определяется полем `type`.

```json
{
  "event_id":    "01913e4f-9b5e-7000-a0d0-cf2f4a0b3ee0",
  "type":        "order.created",
  "version":     1,
  "order_id":    42,
  "user_id":     7,
  "status":      "new",
  "items": [
    {
      "product_id": 101,
      "quantity":   2,
      "price":      1500
    }
  ],
  "occurred_at": "2026-04-16T12:00:00Z"
}
```

| Поле          | Тип      | Описание                                      |
|---------------|----------|-----------------------------------------------|
| `event_id`    | UUID v7  | Уникальный ID события — ключ дедупликации     |
| `type`        | string   | Тип события (см. ниже)                        |
| `version`     | int      | Версия схемы. Текущая: `1`                    |
| `order_id`    | int64    | ID заказа                                     |
| `user_id`     | int64    | ID пользователя                               |
| `status`      | string   | Статус заказа на момент события               |
| `items`       | array    | Позиции заказа                                |
| `occurred_at` | RFC 3339 | Время наступления события (UTC)               |

**items[i]:**

| Поле         | Тип   | Описание              |
|--------------|-------|-----------------------|
| `product_id` | int64 | ID продукта           |
| `quantity`   | int   | Количество (> 0)      |
| `price`      | int64 | Цена в копейках (≥ 0) |

---

### Типы событий

#### `order.created`

Публикуется при успешном создании заказа.

- `status` всегда `"new"`

#### `order.cancelled`

Публикуется при отмене заказа.

- `status` всегда `"cancelled"`

---

## Versioning

При **обратно-совместимых** изменениях (добавление необязательного поля) — `version` не меняется.  
При **breaking changes** — `version` инкрементируется, потребитель обязан проверять поле `version`.

## Delivery guarantee

Используется **Transactional Outbox**: событие записывается в таблицу `outbox` в той же транзакции, что и изменение заказа. Relay читает `outbox` и публикует в Kafka.

Гарантия: **at-least-once**. Потребители должны быть идемпотентны (дедупликация по `order_id` + `type`).
